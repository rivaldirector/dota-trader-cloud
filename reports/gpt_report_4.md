# Dota 2 Quant Betting System — Отчёт #4 для GPT

## Критический баг найден и исправлен

В отчёте #3 все метрики были неверными из-за ошибки матчинга данных.

**Суть бага:** `odds_snapshots.team_1_name` (BetsAPI home team) ≠ `matches.team_1_name` (PandaScore team1) в **64.3% матчей**. Мы считали edge как `adj_prob(PandaScore team1) − market_prob(BetsAPI home team)`, где это разные команды. Анализ был бессмысленным.

**Причина:** функция `_match_to_db` правильно находит PandaScore match_id, но порядок home/away у BetsAPI может быть обратным. Дополнительно в ~5% случаев fuzzy matching связывал BetsAPI матч с неправильным PandaScore матчем (разные команды полностью).

**Исправление в анализе:** перед вычислением mp_open проверяем какая BetsAPI команда соответствует PandaScore team1 по нормализованному имени. Если BetsAPI home = PandaScore team2, инвертируем вероятность. Матчи без подтверждённого выравнивания исключаются.

```python
def get_mp_for_team1(m_t1, m_t2, o_t1, o_t2, odds1, odds2):
    if team_match(m_t1, o_t1) and team_match(m_t2, o_t2):
        return novig(odds1, odds2)          # правильный порядок
    if team_match(m_t1, o_t2) and team_match(m_t2, o_t1):
        p2, p1 = novig(odds1, odds2)
        return p1, p2                        # перевёрнутый — инвертируем
    return None, None, False                 # неизвестно — исключаем
```

## Вопросы к GPT

### 1. Две версии сигнала

**Вариант A (все ставки, n=55):** ROI=+0.700, CI=[+0.253,+1.214]. Высокий ROI из-за высоких коэффициентов. CLV+=0.655.

**Вариант B (только odds<2.0, n=35):** ROI=+0.186, CI=[+0.022,+0.329]. Едва значимый. CLV+=0.743.

Какой вариант более надёжен для paper trading: A (высокий ROI, более широкий CI) или B (маленький ROI, CLV независимо сильнее, нет риска)? Или правило надо заблокировать на odds<2.0 сейчас?

### 2. CLV+ = 0.743 при n=35 — насколько это значимо?

При n=35 и CLV+=0.743, это 26/35 случаев где рынок движется к нам. По биномиальному тесту (p=0.5, n=35, k=26): p-value = ? Является ли это независимым доказательством edge при таком n?

### 3. Отчего negative correlation глобально при положительном CLV+?

Глобально: corr(edge_adj, result) = −0.216, avg_CLV = −0.009.  
В сегменте: corr(edge_adj[seg], result[seg]) = +0.255, avg_CLV[seg] = +0.092.

Почему одна и та же модель с отрицательной глобальной корреляцией имеет положительную корреляцию в сегменте? Это selection bias (мы отбираем только положительные edges) или реальная структурная разница?

### 4. Почему Elo diff ≥ 150 — граница?

При elo_diff < 150 (n=437): corr = −0.216 — модель хуже рынка.  
При elo_diff ≥ 150 (n=101): corr ≈ 0 → +0.292 — модель лучше рынка.

Механически: при elo_diff < 150 Elo недооценивает фаворита (рынок ставит 0.75, Elo 0.60) → edge отрицательный для правильного победителя. При elo_diff ≥ 150 Elo более уверен и переоценивает сильную команду относительно рынка → edge положительный.

Есть ли теоретическое обоснование почему именно 150 пунктов разделяет два режима? Это специфика нашего K=32 и dataset размера, или универсальная граница для Elo?

### 5. Как обращаться с high-odds бетами в сегменте?

Сейчас: при ело_diff≥150 и adj_prob > market_prob, если bet_odds≥2.0 — стратегия ставит на большого underdog'а согласно Elo. CLV+=0.500 (не отличается от случайного). WR=0.550.

Варианты:
- (a) Исключить все ставки с bet_odds ≥ 2.0 (строгий фильтр)
- (b) Оставить, но уменьшить размер ставки пропорционально Kelly
- (c) Требовать дополнительно h2h_n ≥ 2 для high-odds ставок
- (d) Оставить как есть, CLV может улучшиться с ростом n

Что более правильно методологически?
